BUILD ADVANCED ORDERS PHASE 1 ONLY

Implement Phase 1 from the approved advanced-orders plan.

Scope:
Take Profit and Stop Loss orders attached to existing paper positions only.

Do NOT build:

* buy limit orders
* candle-history verification
* background workers
* cron polling
* always-on order scanning
* OCO orders
* ladder exits
* push notifications
* alerts outside in-app toasts

Cost requirement:
No new external market data calls in Phase 1.
Reuse the current price/market cap data already fetched by valuePositions().

Core requirements:

1. Add paper_orders table

* Store pending TP/SL orders
* Link to wallet/user
* Link to token/position
* Support trigger type: market_cap or price
* Support order types: take_profit and stop_loss
* Support amount type: percent_position
* Support amount values: 25, 50, 100
* Status: pending, filled, canceled, failed

2. Create order endpoints

* Create order
* List orders
* Cancel order
* Use existing ownership checks for linked accounts
* Preserve guest mode separately

3. Trigger evaluation

* Run during positions refresh only
* Evaluate pending orders for the current wallet only
* Use current MC/price already loaded for positions
* Take Profit triggers when current MC/price >= target
* Stop Loss triggers when current MC/price <= stop
* Fill through existing executeSell()
* Apply existing slippage/liquidity/P&L/stats logic
* Orders must be idempotent and cannot fill twice

4. Mini Planner integration

* Add Attach Take Profit checkbox
* Add Attach Stop Loss checkbox
* Add sell percentage selector: 25%, 50%, 100%
* Apply To Trade should carry TP/SL settings into the pending plan
* Orders are created only after the user manually confirms the paper buy
* No auto-buy

5. UI

* Show attached orders on positions
* Show status: Pending, Filled, Canceled
* Allow canceling pending orders
* Show filled orders in trade history with labels:
    Filled by Take Profit
    Filled by Stop Loss
* Show non-blocking toast when an order fills

6. Guest mode

* Store guest orders in localStorage
* Evaluate guest TP/SL on refresh/focus using the same current price/quote data
* Guest behavior should mirror signed-in users as closely as possible

7. Safety

* Manual buy/sell remains unchanged
* Portfolio calculations unchanged unless an order fills
* Leaderboard changes only after a filled simulated sell
* Failed order evaluation must not corrupt portfolio
* No duplicate fills
* No API cost increase beyond existing position refresh

QA:

* Create position without orders; verify behavior unchanged
* Create position with TP; refresh after trigger crossed; verify one fill
* Create position with SL; refresh after trigger crossed; verify one fill
* Cancel pending order
* Guest order flow works
* Signed-in order flow works
* Trade history labels correct
* Leaderboard updates only after fill
* Typecheck/build passes

Return:

* files changed
* migration summary
* order execution summary
* API cost confirmation
* safety confirmation
* whether safe to publish
