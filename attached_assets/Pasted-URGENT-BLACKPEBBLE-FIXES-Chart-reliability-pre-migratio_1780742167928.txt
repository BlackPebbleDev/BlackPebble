URGENT BLACKPEBBLE FIXES — Chart reliability + pre-migration token trading

The site is live. Fix only these two issues.

====================================================
1. CHART LOADING / LIFECYCLE BUG
====================================================

Problem:
On mobile, after switching between multiple tokens/pages/charts, charts eventually stop loading. The user has to completely close/restart the browser before charts display again.

Expected behavior:
Charts should reliably load every time a user:
- Switches tokens
- Moves between Trading / Markets / Portfolio / Leaderboard / Utilities
- Opens a token from markets
- Opens a token from positions/history/watchlist
- Returns to a previously viewed token
- Uses mobile Safari, Phantom browser, or in-app webview

Diagnose and fix chart lifecycle.

Requirements:
- Inspect chart component lifecycle.
- Ensure chart iframe/widget fully remounts when token address/pair changes.
- Add stable unique React key based on token pair/address + timeframe if needed.
- Clean up old chart instances, listeners, timers, and iframe references on unmount.
- Prevent stale chart state from blocking new renders.
- Add loading state.
- Add timeout fallback.
- Add “Reload Chart” button if chart fails to load.
- Prevent memory buildup when switching tokens repeatedly.
- Avoid API spam or infinite reload loops.

QA:
- Open at least 10 different tokens in a row.
- Switch between tabs/pages repeatedly.
- Return to previous tokens.
- Refresh page.
- Confirm charts load every time.
- Confirm no console errors.
- Run typecheck/build.

====================================================
2. PRE-MIGRATION / LOW-LIQUIDITY TOKEN TRADING
====================================================

Problem:
Some pre-migration or low-liquidity tokens are blocked from paper trading because liquidity/reserve data is missing, incorrect, extremely low, or unavailable.

Expected behavior:
BlackPebble should allow paper trading on these tokens when enough price and market cap data exists, but with stricter simulation limits.

Diagnose why trades are blocked.

Check:
- trade validation
- liquidity checks
- slippage simulation
- market cap checks
- total supply checks
- token data freshness
- DexScreener fields
- pre-migration token handling

Fallback rules:
If liquidity data is missing or unreliable but token price and market cap are usable:
- Allow small paper trades.
- Apply conservative fallback slippage.
- Apply stricter max trade size.
- Apply stricter max position size.
- Mark internally as high-risk / low-liquidity simulation.
- Prevent leaderboard abuse.

Only block trades when:
- No usable price exists.
- No usable market cap exists.
- Token supply is missing and position cap cannot be calculated.
- Token data is stale.
- Trade exceeds realistic max ownership or max impact limits.

User-facing error messages:
- “Trading unavailable: insufficient live market data.”
- “Order too large for current simulated liquidity.”
- “This token has limited market data. Smaller simulated trades only.”

QA:
- Test normal migrated tokens.
- Test low-liquidity tokens.
- Test pre-migration tokens.
- Test tokens with missing liquidity but usable price/MC.
- Test buy and sell flows.
- Test leaderboard scoring.
- Test position caps.
- Test slippage calculations.
- Verify no unrealistic fills are possible.
